# Формалізація задачі штучного інтелекту в межах інформаційної системи

**Дисципліна:** Проєктування інформаційних систем  
**Практична робота №7**  
**Система:** Платформа для продажу та оренди житла з модулем виявлення фейкових оголошень  
**Технологічна платформа:** MuleSoft Anypoint Platform

---

## 1. Опис проблемної області

### Система

Платформа для розміщення оголошень про продаж і оренду житлової нерухомості. Користувачі можуть публікувати оголошення з описом, фотографіями, ціною та контактними даними. Інші користувачі переглядають оголошення та зв'язуються з продавцями чи орендодавцями.

Система реалізована на базі MuleSoft Anypoint Platform і включає:
- API-шар для мобільного та веб-клієнтів
- Систему модерації оголошень
- Базу даних оголошень і профілів користувачів
- Інтеграцію з зовнішніми сервісами (геокодування, телефонна верифікація)

### Межі системи

| Компонент | Входить до системи | Поза межами |
|---|---|---|
| API для прийому оголошень | Так | — |
| ML-модуль класифікації | Так | — |
| База даних оголошень | Так | — |
| Черга обробки Anypoint MQ | Так | — |
| Сторонній сервіс верифікації паспорта | — | Так |
| Зовнішні банківські API | — | Так |

### Стейкхолдери

- **Користувач (покупець/орендар)** — переглядає оголошення, очікує достовірної інформації.
- **Користувач (продавець/орендодавець)** — розміщує оголошення, очікує швидкої публікації.
- **Оператор модерації** — отримує підозрілі оголошення для ручної перевірки.
- **Адміністратор системи** — контролює якість роботи ML-моделі, налаштовує пороги.
- **Бізнес (платформа)** — зацікавлений у довірі користувачів і зниженні кількості скарг.

## 2. Формалізація задачі штучного інтелекту

### 2.1 Бізнес-задача

> **Автоматично виявляти фейкові та шахрайські оголошення ще до їх публікації, щоб захистити користувачів від шахрайства та зберегти репутацію платформи.**

Фейкові оголошення — це оголошення з навмисно неправдивою інформацією: неіснуючий об'єкт, завищена/занижена ціна як приманка, підроблені фотографії або контактні дані.

### 2.2 Перетворення на задачу штучного інтелекту

| Рівень | Формулювання |
|---|---|
| Бізнес-задача | Не допустити публікації фейкових оголошень |
| Задача ШІ | За ознаками оголошення та профілю автора передбачити, чи є оголошення фейковим |
| Формальна постановка | Бінарна класифікація: `f(X) → {0: справжнє, 1: фейк}` |

### 2.3 Тип задачі: Класифікація

**Обраний тип:** Бінарна класифікація.

**Обґрунтування:**
- Результат — дискретна мітка з двох класів: `фейк` / `не фейк`.
- Система має відносити кожне оголошення до одного з наперед визначених класів.

**Чому не регресія:**
- Регресія передбачає числовий неперервний вихід (наприклад, ціна).
- Нам не потрібно передбачати числове значення — потрібне рішення "так/ні".

**Чому не рекомендація:**
- Рекомендація пропонує найбільш підхожий об'єкт з набору.
- Тут не відбувається підбір оголошень — відбувається оцінка конкретного оголошення.

**Розширення (опційно):** Додатково можна виводити `confidence score` (0.0–1.0) — це числовий результат класифікатора, що дозволяє гнучко налаштовувати поріг блокування. Це не перетворює задачу на регресію — confidence є допоміжним, а не цільовим виходом.

## 3. Опис вхідних та вихідних даних

### 3.1 Вхідні атрибути

| Атрибут | Тип | Джерело |
|---|---|---|
| Текст заголовку оголошення | string | Форма подачі |
| Текст опису | string | Форма подачі |
| Ціна (грн) | float | Форма подачі |
| Кількість фотографій | int | Форма подачі |
| Геолокація (місто, район) | string/geo | Форма подачі |
| Площа об'єкта (м²) | float | Форма подачі |
| Кількість кімнат | int | Форма подачі |
| Час доби подачі | timestamp | Система |
| Кількість оголошень від цього користувача за останні 24 год | int | БД профілів |
| Вік облікового запису (дні) | int | БД профілів |
| Кількість попередніх скарг на користувача | int | БД модерації |
| Хеш фотографій (перевірка дублікатів) | hash[] | Сервіс аналізу зображень |
| Телефонний номер (верифікований?) | bool | Телефонна верифікація |
| IP-адреса (регіон) | string | Система |

### 3.2 Джерела даних

- **Пряме введення:** форма подачі оголошення (API).
- **Системна метаінформація:** час, IP, fingerprint пристрою.
- **Профіль користувача:** агрегати з БД.
- **Зовнішня верифікація:** телефонний сервіс.
- **Сервіс аналізу зображень:** перевірка на дублікати та стокові фото.

### 3.3 Очікуваний вихід моделі

```
{
  "listing_id": "uuid",
  "verdict": "FAKE" | "LEGITIMATE",
  "confidence": 0.0 - 1.0,
  "recommended_action": "BLOCK" | "MANUAL_REVIEW" | "PUBLISH"
}
```

### 3.4 Користь для бізнесу та користувача

| Стейкхолдер | Користь |
|---|---|
| Покупець/орендар | Бачить лише перевірені оголошення; менше шахрайства |
| Продавець | Чесні продавці не конкурують із фейковими ціновими приманками |
| Оператор | Обробляє лише ті випадки, де впевненість моделі низька |
| Бізнес | Зниження скарг, зростання довіри, зменшення репутаційних ризиків |

## 4. Архітектурне рішення для обробки подій і даних

### 4.1 Потреба в асинхронній обробці

Так. Після отримання оголошення через API **не можна** синхронно чекати результату ML-моделі (затримка 200–2000 мс) у тому самому HTTP-запиті. Користувач має отримати миттєве підтвердження "оголошення прийнято на перевірку", а обробка відбувається асинхронно.

### 4.2 Обрані інструменти

#### Anypoint MQ — основна черга нових оголошень

**Причина:** Anypoint MQ забезпечує надійну асинхронну взаємодію між розподіленими компонентами — між API-шаром (producer) та ML-сервісом класифікації (consumer). Оголошення не буде втрачено навіть у разі тимчасової недоступності ML-сервісу (at-least-once delivery).

**Використання:** Кожне нове оголошення публікується в чергу `listing.new`. ML-worker читає з черги та обробляє.

#### Object Store — стан сесії перевірки

**Причина:** Під час обробки оголошення є проміжний стан: "прийнято → в обробці → рішення моделі → очікує оператора". Цей стан потрібно зберігати між кроками розподіленого процесу. Object Store — нативний механізм MuleSoft для зберігання проміжних даних процесу.

**Використання:** Зберігає `{ listing_id, stage, partial_results, timestamp }` між кроками потоку.

## 5. Схема рішення

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         ДЖЕРЕЛО ПОДІЇ / ДАНИХ                               │
│                                                                             │
│   [Веб/Мобільний клієнт]                                                   │
│         │  POST /listings                                                   │
└─────────┼───────────────────────────────────────────────────────────────────┘
          │
          ▼
┌─────────────────────┐
│   MuleSoft API GW   │  ← Валідація схеми, автентифікація, rate limiting
│  (Experience API)   │
└─────────┬───────────┘
          │ Publish
          ▼
┌─────────────────────┐     ┌──────────────────────┐
│   Anypoint MQ       │     │    Object Store       │
│  listing.new queue  │     │  Стан: "IN_QUEUE"     │
└─────────┬───────────┘     └──────────────────────┘
          │ Consume
          ▼
┌─────────────────────────────────────────────────────────────┐
│              ML Classification Worker (MuleSoft Flow)       │
│                                                             │
│  1. Зчитати дані оголошення                                 │
│  2. Звернутись до ML-моделі (HTTP / Python service)         │
│  3. Отримати verdict + confidence                           │
│  4. Оновити Object Store → стан "CLASSIFIED"                │
└─────────┬───────────────────────────────────────────────────┘
          │
          ├─── confidence > 0.7 AND verdict = FAKE
          │         ▼
          │    [Автоблокування]
          │    Оголошення → статус BLOCKED у БД
          │    Сповіщення користувача
          │
          ├─── confidence 0.4–0.7 (сіра зона)
          │         ▼
          │    ┌────────────────┐
          │    │  Anypoint MQ   │
          │    │  (listing.rev) │
          │    └────────┬───────┘
          │             │
          │             ▼
          │       [Оператор]
          │    Ручна модерація → BLOCK або PUBLISH
          │
          └─── confidence < 0.4 AND verdict = LEGITIMATE
                    ▼
              [Автопублікація]
              Оголошення → статус PUBLISHED у БД
              Доступне для перегляду користувачами
```

## 6. Обмеження і ризики

| Ризик | Опис | Мітигація |
|---|---|---|
| **Хибно-позитивні результати** | Модель блокує справжнє оголошення | Поріг автоблокування встановлено на confidence > 0.85; нижче — ручна перевірка |
| **Хибно-негативні результати** | Фейк проходить як справжній | Постмодерація за скаргами; periodical re-scoring |
| **Втрата повідомлень** | Збій між API і ML-worker | Anypoint MQ — at-least-once delivery; дедублікація за `listing_id` |
| **Порушення порядку обробки** | Старі підозрілі оголошення висять довше нових | FIFO Queue для черги модерації — строгий порядок |
| **Кешування застарілих даних репутації** | Заблокований юзер отримав кеш "чистого" профілю | TTL кешу ≤ 600 с; при блокуванні — примусова інвалідація кешу (`DEL cache.user_reputation:{id}`) |
| **Неможливість кешування** | Фотографії та персональні дані не можна кешувати безконтрольно | Фото-хеші — так; самі фото — ні. GDPR: PII (телефон, IP) — зберігати з TTL, не в Redis |
| **Модель застаріває** | Шахраї адаптуються до моделі | Моніторинг метрик precision/recall; scheduled retraining |
| **Перевантаження черги** | Пік публікацій (ранок, вихідні) | Горизонтальне масштабування ML-worker; dead-letter queue для незброблених повідомлень |
| **Критичний порядок** | При апеляції від заблокованого користувача важливий порядок — хто подав скаргу першим | FIFO Queue для черги апеляцій |
| **Втручання людини** | При confidence 0.4–0.7 обов'язкова ручна перевірка | SLA оператора — до 4 годин; якщо перевищено — автопублікація з позначкою "не перевірено повністю" |

## 7. Таблиця формалізації задачі ШІ

| Поле | Значення |
|---|---|
| **Бізнес-задача** | Виявлення та блокування фейкових оголошень на платформі нерухомості |
| **Тип задачі ШІ** | Бінарна класифікація |
| **Вхід** | Текст, ціна, фото-хеші, геолокація, метадані профілю користувача, час подачі, телефонна верифікація |
| **Вихід** | `verdict: FAKE/LEGITIMATE`, `confidence: 0.0–1.0`, `recommended_action: BLOCK/MANUAL_REVIEW/PUBLISH` |
| **Користь** | Захист користувачів від шахрайства; зростання довіри до платформи; зниження навантаження на модераторів |
| **Обране архітектурне рішення** | Anypoint MQ (надійна асинхронна черга нових оголошень), Object Store (проміжний стан перевірки) |
| **Обмеження** | модель потребує periodical retraining; при confidence 0.4–0.7 обов'язкове втручання людини |
